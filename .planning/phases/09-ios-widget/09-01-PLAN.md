<task type="auto">
  <name>Task 1: Install @bacons/apple-targets and create widget target</name>
  <files>package.json, app.json, targets/widget/expo-target.config.js, targets/widget/index.swift, targets/widget/Assets.xcassets/</files>
  <action>
  1. Run `npx create-target widget` to scaffold the widget target directory and install @bacons/apple-targets. This creates targets/widget/ with boilerplate files.

  2. Configure targets/widget/expo-target.config.js:
     - type: "widget"
     - name: "QuickLaunch"
     - frameworks: ["SwiftUI"]
     - deploymentTarget: "16.0" (required for lock screen widgets)
     - colors: accent color (#4A90D9 matching app theme), widget background (light #FFFFFF / dark #1C1C1E)
     - Do NOT configure App Groups (not needed for launch-only widget)

  3. Update app.json:
     - Add "@bacons/apple-targets" to plugins array (if not already added by create-target)
     - Add ios.appleTeamId — check user_setup, prompt user for their Team ID if not known
     - Keep existing plugins (expo-router, expo-splash-screen, expo-sqlite)

  4. Write targets/widget/index.swift with SwiftUI widget code:
     - QuickLaunchEntry (TimelineEntry with just date)
     - QuickLaunchProvider (TimelineProvider with .never refresh policy — static widget)
     - QuickLaunchWidgetView with two families:
       a. .systemSmall (home screen): Show brain icon, "ADHD Needs" title, "What do I need?" subtitle, using accent and widget background colors from Assets
       b. .accessoryCircular (lock screen): Just the brain icon with AccessoryWidgetBackground
     - Use @Environment(\.widgetFamily) to switch between layouts
     - Set .widgetURL(URL(string: "adhdneeds://")) on both views — this opens the app to home screen (craving selection) when tapped
     - @main Widget struct supporting [.systemSmall, .accessoryCircular] families
     - configurationDisplayName: "ADHD Needs"
     - description: "Quickly identify what you really need right now."

  5. Set up widget Assets.xcassets with color sets for $accent and $widgetBackground (light/dark variants).

  AVOID:
  - Do NOT use App Groups — not needed for launch-only widget, adds unnecessary complexity
  - Do NOT use .accessoryRectangular or .systemMedium — keep it minimal (small + circular only)
  - Do NOT use ESM syntax in expo-target.config.js — must be CommonJS (module.exports)
  - Do NOT run expo prebuild yet — that's Task 2
  </action>
  <verify>
  - targets/widget/expo-target.config.js exists with correct config
  - targets/widget/index.swift exists with SwiftUI widget code
  - app.json has @bacons/apple-targets in plugins
  - package.json has @bacons/apple-targets in dependencies
  </verify>
  <done>Widget target scaffolded, Swift code written, Expo plugin configured</done>
</task>

<task type="auto">
  <name>Task 2: Generate native project and verify iOS build</name>
  <files>ios/</files>
  <action>
  1. Run `npx expo prebuild -p ios --clean` to generate the native iOS project with widget extension target.

  2. Verify the generated ios/ directory contains:
     - The main app target
     - A widget extension target (QuickLaunch)
     - Proper Swift files linked from targets/widget/

  3. Attempt to build the iOS project:
     - Run `xcodebuild -workspace ios/adhdneeds.xcworkspace -scheme adhdneeds -sdk iphonesimulator -configuration Debug build` (or similar — adapt to actual workspace/scheme names)
     - If build fails, diagnose and fix issues (common: missing frameworks, deployment target mismatch, signing issues)

  4. If appleTeamId was not set (user hasn't provided it yet), the build may fail on code signing. This is expected — the checkpoint in Task 3 will handle this. Focus on verifying the widget extension is correctly included in the project structure.

  AVOID:
  - Do NOT modify files inside ios/ directly — all changes go through targets/widget/ and app.json
  - Do NOT force-clean Xcode derived data unless build fails with cache issues
  </action>
  <verify>
  - ios/ directory exists after prebuild
  - Widget extension target appears in Xcode project
  - Build completes (or only fails on code signing, not on Swift compilation)
  </verify>
  <done>Native iOS project generated with widget extension, Swift code compiles</done>
</task>
